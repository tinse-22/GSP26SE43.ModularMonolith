## 1. Tổng quan Workflow (End-to-End)

```
[User]
  │
  ├─ 1. CREATE PROJECT          POST /projects
  ├─ 2. UPLOAD API SPEC         POST /projects/{id}/specifications/upload (multipart)
  │       └─ Poll parseStatus   GET  /projects/{id}/specifications/{specId}
  ├─ 3. VIEW ENDPOINTS          GET  /projects/{id}/specifications/{specId}/endpoints
  ├─ 4. CREATE TEST SUITE       POST /projects/{id}/test-suites
  │       ├─ Propose order      POST /test-suites/{id}/order-proposals
  │       └─ Approve order      POST /test-suites/{id}/order-proposals/{pid}/approve
  ├─ 5. GENERATE LLM SUGGESTIONS
  │       ├─ Generate           POST /test-suites/{id}/llm-suggestions/generate
  │       ├─ List               GET  /test-suites/{id}/llm-suggestions
  │       └─ Review (per item)  PUT  /test-suites/{id}/llm-suggestions/{sid}/review
  ├─ 6. GENERATE TEST CASES     POST /test-suites/{id}/generate-tests
  │       └─ Poll job status    GET  /test-suites/{id}/generation-status?jobId=
  ├─ 7. EXECUTE TEST RUN        POST /test-suites/{id}/test-runs
  │       └─ Real-time updates  SignalR: TestRunStatusChanged
  ├─ 8a. ON FAILURE BRANCH
  │       ├─ Auto trigger       useAutoLLMAnalysis hook (SignalR-triggered)
  │       └─ Explain failure    POST /test-suites/{id}/test-runs/{rid}/failures/{cid}/explanation
  ├─ 8b. DETERMINISTIC BRANCH   GET  /test-suites/{id}/test-runs/{rid}/results
  └─ 9. EXPORT REPORT           POST /test-suites/{id}/test-runs/{rid}/reports
          └─ Download           GET  /test-suites/{id}/test-runs/{rid}/reports/{repId}/download
```

#### ⭐ Câu hỏi đặc biệt: Khi user nhập `requirements` / `business rules`

**FE KHÔNG gọi API ngay khi user nhập**.

- **`GlobalBusinessRules`**: User nhập vào textarea trong Create Test Suite modal → FE lưu vào **form state (local)** → submit cùng body `POST /projects/{projectId}/test-suites` tại bước 4.

**Dữ liệu được dùng tiếp trong flow**: `GlobalBusinessRules` và `EndpointBusinessContexts` được BE dùng như context input khi LLM generate suggestions ở Step 5. FE không cần truyền lại — BE đọc từ suite record.

### STEP 8a — On Failure Branch: LLM Failure Explanation

**Trigger**: Tự động qua `useAutoLLMAnalysis` hook sau khi SignalR emit `TestRunStatusChanged` với `status=Completed/Failed`

**Hook flow** (`useAutoLLMAnalysis`):
1. Nhận `TestRunStatusChanged` event từ SignalR
2. Kiểm tra `processedRuns` trong `localStorage["autoLLMAnalysis_processedRuns"]` — skip nếu đã xử lý
3. Fetch run detail: `GET /test-suites/{suiteId}/test-runs/{runId}/results`
4. Filter các case có `status === "Failed"`
5. Concurrent (max 5 parallel): gọi `explainFailure` cho từng case
6. Cập nhật state `explanationsProgress` theo từng case hoàn thành

**Failure Explanation API**:

| Field | Value |
|---|---|
| **Fetch existing** | `GET /test-suites/{suiteId}/test-runs/{runId}/failures/{testCaseId}/explanation` |
| **Generate new** | `POST /test-suites/{suiteId}/test-runs/{runId}/failures/{testCaseId}/explanation` |
| **Request body (POST)** | `{}` (context được BE tự lấy từ run/case data) |

**Response `FailureExplanationModel`**:
```json
{
  "testSuiteId": "string",
  "testRunId": "string",
  "testCaseId": "string",
  "summaryVi": "Tóm tắt lỗi bằng tiếng Việt",
  "possibleCauses": ["Nguyên nhân 1", "Nguyên nhân 2"],
  "suggestedNextActions": ["Action 1", "Action 2"]
}
```

**FE hiển thị**: Panel nổi (floating panel) tại `TestRunsPage` hoặc `SuggestionsPage`:
- Tên test case
- `summaryVi` — tóm tắt lỗi
- `possibleCauses[]` — accordion list
- `suggestedNextActions[]` — checklist

**Hook state**:
```ts
{
  isRunning: boolean,
  currentRunId: string | null,
  currentSuiteId: string | null,
  suggestionsStatus: "idle" | "loading" | "success" | "error",
  explanationsStatus: "idle" | "loading" | "partial" | "complete" | "error",
  explanationsProgress: { completed: number, total: number },
  explanationsErrors: Record<testCaseId, string>
}
```

**Manual flow** (SuggestionsPage):
1. User chọn suite → `GET /test-suites/{suiteId}/test-runs`
2. User chọn run → `GET /test-suites/{suiteId}/test-runs/{runId}/results` (retry 4× nếu `resultsSource=unavailable`)
3. "Generate All" → parallel generate cho tất cả failed cases

## 3. Nhánh xử lý

### Nhánh Success
```
Test Run completed → resultsSource === "live"
→ FE loads results
→ passRate ≥ threshold → show green badge
→ No failed cases → useAutoLLMAnalysis skips explanation step
→ User generates report → download
```

### Nhánh Failure
```
Test Run completed → failedCount > 0
→ SignalR TestRunStatusChanged received
→ useAutoLLMAnalysis: check processedRuns cache
→ Fetch run results → filter failed cases
→ Concurrent (max 5): POST /failures/{caseId}/explanation
→ Display floating panel với summaryVi + possibleCauses + suggestedNextActions
→ User có thể dismiss panel (lưu dismissedRuns vào localStorage)
→ User có thể retry failed explanations
```

### Nhánh Deterministic
```
User mở TestRunsPage → chọn run → expand
→ GET /test-runs/{runId}/results
→ FE đọc từng TestCaseRunDetail:
  - statusCodeMatched, schemaMatched, bodyContainsPassed,
    jsonPathChecksPassed, responseTimePassed
→ Hiển thị per-check indicators (không cần LLM)
→ failureReasons[] hiển thị trực tiếp text lỗi từ BE
```

### Nhánh Review / Approve / Reject (LLM Suggestions)

```
[Pending suggestion]
    ├─ User click Approve
    │   → PUT /llm-suggestions/{id}/review { action: "Approve", rowVersion }
    │   → Suggestion.reviewStatus = "Approved"
    │   → Counted toward generation threshold
    │
    ├─ User click Reject
    │   → PUT /llm-suggestions/{id}/review { action: "Reject", rowVersion, reviewNotes? }
    │   → Suggestion.reviewStatus = "Rejected"
    │   → Not included in generate-tests
    │
    ├─ User click Modify
    │   → PUT /llm-suggestions/{id}/review { action: "Modify", rowVersion, modifiedContent: {...} }
    │   → Suggestion.reviewStatus = "Modified"
    │   → modifiedContent replaces original
    │
    └─ Bulk Actions
        → POST /llm-suggestions/bulk-review
        → { action, filterByTestType?, filterByEndpointId? }
```

## 5. Contract để BE đối chiếu FE

### 5.1 Fields bắt buộc FE phải truyền đúng

| API | Field bắt buộc | Ghi chú |
|---|---|---|
| POST /test-suites | `Name` (PascalCase!) | FE dùng PascalCase keys |
| POST /test-suites | `ApiSpecId` | ID của spec đã parse |
| POST /test-suites | `SelectedEndpointIds[]` | Phải là UUID hợp lệ từ BE |
| POST /test-suites | `GenerationType` | Enum string: "Auto" \| "Manual" \| "LLMAssisted" |
| PUT /llm-suggestions/.../review | `rowVersion` | Optimistic concurrency, phải match BE |
| PUT /llm-suggestions/.../review | `action` | Chính xác "Approve" \| "Reject" \| "Modify" |
| POST /test-runs | `environmentId` | UUID của environment, không được null |
| POST /test-runs | `strictValidation` | Boolean, không thiếu |
| POST /order-proposals/.../approve | `RowVersion` | Từ response của proposeOrder |
| POST /specifications/upload | `SourceType` | Số: 0, 1, hoặc 2 — không phải string |
| POST /llm-suggestions/generate | `algorithmProfile` | Object đầy đủ 5 boolean fields |

### 5.2 Dấu hiệu FE đã nối đúng flow

| Dấu hiệu | Cách BE kiểm tra |
|---|---|
| FE gửi PascalCase trong body | Body keys là `Name`, `ApiSpecId`, không phải `name`, `apiSpecId` |
| FE poll parse status đúng | Spec `parseStatus` chuyển Pending → Success trước khi FE call create-suite |
| FE auto propose+approve order sau create suite | `order-proposals` record tồn tại với `status=Approved` ngay sau create-suite |
| FE gửi `algorithmProfile` đầy đủ | LLM suggestions được generate với đúng 5 flags |
| FE dùng `rowVersion` từ response trước | Không bị 409 Conflict trên review |
| FE poll generation status | Job record thấy nhiều GET trước khi Completed |
| FE subscribe SignalR đúng runId | `SubscribeToTestRun(runId)` được gọi sau startTestRun |
| FE check processedRuns cache | Cùng runId không generate explanation 2 lần |
| FE retry getTestRunResults khi unavailable | Thấy nhiều GET trên endpoint results trước khi data available |

### 5.3 State/payload BE nên kiểm tra

```
1. TestSuite.selectedEndpointIds  → phải match endpoints trong spec
2. TestSuite.endpointBusinessContexts → phải là object, keys là endpointId
3. LLMSuggestion.reviewStatus → phải là "Approved" trước khi generate-tests
4. OrderProposal.status → phải là "Approved" trước khi execute
5. TestRun.environmentId → phải resolve đến environment của đúng projectId
6. FailureExplanation.testCaseId → phải thuộc testRunId được chỉ định
```

## 6. Câu hỏi đặc biệt — Requirements & Business Rules

### Khi user nhập `requirements` / `business rules`:

**FE KHÔNG gọi API ngay lập tức.**

| Trường | Nơi user nhập | FE lưu | Khi nào submit | Submit lên API nào |
|---|---|---|---|---|
| `GlobalBusinessRules` | Create Test Suite modal | Local form state | Khi submit tạo suite | `POST /projects/{id}/test-suites` trong field `GlobalBusinessRules` |
| `EndpointBusinessContexts` | Per-endpoint input trong modal | `Record<endpointId, string>` in form state | Khi submit tạo suite | `POST /projects/{id}/test-suites` trong field `EndpointBusinessContexts` |
| Cả hai (update) | TestSuiteDetailPage form | Component state | Click Save | `PUT /projects/{id}/test-suites/{sid}` |

**Dữ liệu được dùng tiếp như thế nào**:
- BE đọc `GlobalBusinessRules` và `EndpointBusinessContexts` từ suite record khi LLM generate suggestions
- FE không cần truyền lại khi gọi `/llm-suggestions/generate` — chỉ cần `specificationId` và `algorithmProfile`
- BE tự join suite context + spec content để build LLM prompt

## 7. Assumptions

1. **LLM backend**: FE có `geminiService.ts` gọi trực tiếp Gemini API (model `gemini-3-flash-preview`) cho `explainFailure` và `suggestExecutionOrder`. Đây có thể là fallback khi BE endpoint chưa sẵn sàng, hoặc chạy song song. **BE cần xác nhận** endpoint `/failures/{id}/explanation` có phải là proxy qua Gemini không.

2. **Generation status polling**: FE poll `/generation-status?jobId=` mỗi 3s. Nếu BE dùng webhook/SignalR thay vì polling, FE cần update `GeneratingTestCasesPage`.

3. **Report formats**: FE request `format: "PDF" | "JSON" | "Markdown"`. FE tải về dưới dạng Blob. Format nội dung JSON/Markdown trong report là assumption dựa trên BE logic.

4. **pageSize=9999 cho endpoints**: FE không dùng server-side pagination cho endpoints. Nếu dataset lớn, đây là bottleneck.

5. **`resultsSource` field**: FE có logic retry 4× khi `resultsSource === "unavailable"`. BE cần đảm bảo field này tồn tại trong response của `/test-runs/{id}/results`.
