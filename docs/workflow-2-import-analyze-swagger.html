Seven Core Workflow
Workflow 2

Import and Analyze Swagger / OpenAPI

Đưa API contract vào hệ thống và biến nó thành bản đồ endpoint có thể dùng để tạo test

Workflow 2 là bước hệ thống đọc Swagger/OpenAPI để hiểu API đang có những endpoint nào, mỗi endpoint cần truyền tham số gì, trả về dữ liệu ra sao và dùng cơ chế bảo mật nào. Đây là nguồn sự thật kỹ thuật giúp các workflow sau sinh test case đúng với API thật.

Swagger
OpenAPI Paths, methods, parameters, schemas, responses và security rules.

Overview nhanh

Mục tiêu

Chuyển file Swagger/OpenAPI từ tài liệu kỹ thuật thành danh sách endpoint rõ ràng, có thể phân tích, ghép với SRS và dùng để generate test.

Đầu vào

File JSON/YAML, URL Swagger, version API, project đang chọn và environment đã cấu hình ở Workflow 1.

Đầu ra

API catalog gồm endpoint, method, request body, query/path parameters, response schema, status code và security requirement.

Các bước chạy chuẩn của Workflow 2

Chọn nguồn Swagger

Người dùng upload file Swagger/OpenAPI hoặc nhập URL tài liệu API để hệ thống lấy nội dung về phân tích.

Kiểm tra định dạng

Hệ thống xác định tài liệu có đọc được không, đúng JSON/YAML không và có đủ cấu trúc OpenAPI cơ bản không.

Đọc danh sách endpoint

Các path và HTTP method như GET, POST, PUT, PATCH, DELETE được tách thành từng API action riêng biệt.

Phân tích tham số request

Hệ thống nhận diện path parameter, query parameter, header parameter và request body để biết mỗi request cần dữ liệu gì.

Phân tích response

Các status code, response body và schema trả về được ghi nhận để sau này dùng làm expectation khi chạy test.

Đọc security scheme

Hệ thống xác định endpoint nào cần bearer token, API key, basic auth hoặc cơ chế xác thực khác.

Chuẩn hóa API catalog

Thông tin endpoint được sắp xếp lại thành một catalog thống nhất, dễ tra cứu và dễ nối với requirement từ SRS.

Lưu kết quả phân tích

Khi hoàn tất, hệ thống lưu metadata API để Workflow 4, Workflow 5 và Workflow 6 có thể sử dụng trực tiếp.

Bên trong workflow hoạt động ra sao?

1. FE upload hoặc tạo spec

SpecificationPage gọi API upload/manual/curl-import, gửi file OpenAPI hoặc dữ liệu endpoint thủ công kèm project, tên và version.

2. Tạo bản ghi specification

Backend lưu ApiSpecification ở trạng thái Pending, giữ file gốc trong Storage và chuẩn bị job parse để xử lý nội dung.

3. Parser đọc contract

ParseUploadedSpecificationCommand chọn parser phù hợp, đọc paths, methods, parameters, request body, responses và security schemes.

4. Ghi endpoint metadata

Kết quả parse được chuẩn hóa rồi lưu thành ApiEndpoint, EndpointParameter, EndpointResponse, EndpointSecurityReq và SecurityScheme.

5. Cấp API catalog

Khi parse thành công, API catalog này trở thành nguồn dữ liệu cho order proposal, merge SRS, parameter mapping và generation.

Luồng code và bảng dữ liệu

  • FE: SpecificationPage dùng specificationService gọi upload/manual/curl-import dưới route /api/projects/{projectId}/specifications.
  • Controller: SpecificationsController dispatch UploadApiSpecificationCommand, CreateManualSpecificationCommand hoặc ImportCurlCommand.
  • Persist spec: metadata của tài liệu được lưu vào apidoc.ApiSpecifications; project scope nằm trong apidoc.Projects.
  • Parser: ParseUploadedSpecificationCommandHandler download file, chọn ISpecificationParser rồi parse paths, methods, params, responses và security.
  • Persist catalog: kết quả parse ghi vào apidoc.ApiEndpoints, apidoc.EndpointParameters, apidoc.EndpointResponses, apidoc.EndpointSecurityReqs và apidoc.SecuritySchemes.
Bước Bảng được ghi/cập nhật Dữ liệu nằm trong bảng
Chọn project trước khi import apidoc.Projects Project chứa API catalog: OwnerId, Name, BaseUrl, Status và ActiveSpecId nếu spec mới được activate.
Upload file Swagger/OpenAPI classifiedads_storage.FileEntries File gốc được Storage module giữ lại với tên file, content type, dung lượng, owner và category ApiSpec. ID file này được đưa về ApiSpecifications.OriginalFileId.
Tạo bản ghi specification apidoc.ApiSpecifications Một dòng đại diện cho tài liệu import: ProjectId, OriginalFileId, Name, SourceType, Version, ParseStatus, ParsedAt, ParseErrors, IsActive.
Parse từng path + HTTP method apidoc.ApiEndpoints Mỗi operation trong Swagger thành một endpoint con theo ApiSpecId, lưu HttpMethod, Path, OperationId, Summary, Description, Tags, IsDeprecated.
Parse input của endpoint apidoc.EndpointParameters Mỗi path/query/header/body/cookie parameter thành một dòng theo EndpointId, lưu Name, Location, DataType, Format, ContentType, IsRequired, DefaultValue, Schema, Examples.
Parse output của endpoint apidoc.EndpointResponses Mỗi status code/response thành một dòng theo EndpointId, lưu StatusCode, Description, Schema, Examples và Headers.
Parse security scheme toàn spec apidoc.SecuritySchemes Cấu hình auth được khai báo trong Swagger theo ApiSpecId: tên scheme, loại Http/ApiKey/OAuth2/OpenIdConnect, bearer format, vị trí API key, parameter name và config JSON.
Parse security requirement từng endpoint apidoc.EndpointSecurityReqs Endpoint nào cần auth gì sẽ có dòng theo EndpointId, lưu SecurityType, SchemeName và OAuth scopes dạng JSON.

Vai trò trong luồng tổng thể

Workflow 2 là bản đồ kỹ thuật của hệ thống API. Workflow 1 cho biết gọi API ở đâu, còn Workflow 2 cho biết API có những cổng nào, cần truyền dữ liệu gì và phản hồi sẽ có hình dạng ra sao. Nếu thiếu bước này, hệ thống không có nền tảng chắc chắn để sinh test tự động.
Swagger không tự nói hết nghiệp vụ. Nó mô tả cấu trúc API, nhưng chưa giải thích đầy đủ rule kinh doanh. Vì vậy, kết quả của Workflow 2 sẽ tiếp tục được ghép với SRS ở Workflow 4 để tạo test case có ý nghĩa hơn.

Điều kiện để xem là import và phân tích đúng

  • Swagger/OpenAPI đọc được, đúng định dạng và không bị thiếu cấu trúc chính.
  • Các endpoint quan trọng xuất hiện đầy đủ trong API catalog.
  • Path, query, header và body parameters được nhận diện rõ ràng.
  • Response schema và status code được ghi nhận để phục vụ validation.
  • Security requirement được xác định để liên kết với environment ở Workflow 1.
  • Nếu Swagger lỗi thời, test case sinh ra có thể không khớp API thật.
  • Nếu thiếu response schema, hệ thống khó kiểm tra dữ liệu trả về có đúng không.
  • Nếu security scheme bị bỏ sót, request có thể fail vì thiếu auth dù tham số khác đúng.
  • Nếu parameter bị hiểu sai vị trí, dữ liệu có thể bị truyền nhầm vào query, path hoặc body.
  • Nếu API catalog không được lưu theo version, việc so sánh thay đổi API sẽ khó kiểm soát.
Workflow 2 biến Swagger/OpenAPI thành API catalog. Từ catalog này, hệ thống có thể hiểu cấu trúc API, nối API với yêu cầu nghiệp vụ, giải thích tham số và sinh test case có cơ sở.
