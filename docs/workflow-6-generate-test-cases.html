Seven Core Workflow
Workflow 6

Generate Test Cases

Sinh test case từ API contract, yêu cầu nghiệp vụ và cách truyền tham số đã được làm rõ

Workflow 6 là bước hệ thống biến toàn bộ ngữ cảnh đã chuẩn bị thành test case cụ thể. Thay vì chỉ tạo request mẫu, hệ thống dùng Swagger, SRS, environment và parameter mapping để sinh test có mục tiêu: kiểm tra happy path, dữ liệu biên, lỗi nghiệp vụ, quyền truy cập và các tình huống phụ thuộc giữa nhiều request.

Test Case
Generation Happy path, negative, boundary, security, dependency và expected result.

Overview nhanh

Mục tiêu

Tạo bộ test case có thể chạy được, có dữ liệu đầu vào, expectation rõ ràng và liên kết được với endpoint/requirement.

Đầu vào

Environment, API catalog, SRS analysis, merge context, parameter mapping, test scope và loại test người dùng muốn sinh.

Đầu ra

Danh sách test case gồm request, test data, expected status, assertions, dependency, priority và covered requirement IDs.

Các bước chạy chuẩn của Workflow 6

Chọn phạm vi sinh test

Người dùng chọn project, module, endpoint, requirement hoặc test suite muốn sinh test case.

Chọn loại test

Có thể chọn happy path, negative case, boundary case, security case, dependency flow hoặc sinh toàn bộ gợi ý.

Gom context cần thiết

Hệ thống lấy Swagger, SRS, kết quả merge, environment và parameter mapping để tạo bức tranh đầy đủ trước khi sinh test.

Xác định mục tiêu kiểm thử

Mỗi test case được gắn với một mục tiêu rõ: kiểm tra rule nào, endpoint nào, trạng thái nào hoặc lỗi nào.

Tạo dữ liệu đầu vào

Hệ thống đề xuất path/query/body/header values phù hợp, bao gồm dữ liệu hợp lệ, dữ liệu sai và dữ liệu biên.

Tạo expected result

Mỗi test case có expected status, điều kiện kiểm tra response, schema hoặc rule nghiệp vụ cần pass/fail.

Sắp xếp dependency

Nếu test cần dữ liệu từ bước trước, hệ thống gắn quan hệ phụ thuộc để thứ tự chạy hợp lý và dễ resolve tham số.

Lưu và chờ review

Test case được lưu lại để người dùng review, chỉnh sửa, approve rồi chuyển sang Workflow 7 để execute.

Bên trong workflow hoạt động ra sao?

1. FE trigger generation

GeneratingTestCasesPage gọi POST generate-tests; backend chỉ nhận khi suite đã có approved order proposal.

2. Tạo generation job

GenerateTestCasesCommand tạo TestGenerationJob trạng thái Queued, trả jobId cho FE và đẩy message vào hàng đợi.

3. Background build payload

TriggerTestGenerationConsumer load job, gọi TestGenerationPayloadBuilder để gom suite, ordered endpoints, SRS context và callback URL.

4. Gọi n8n/LLM

Job chuyển sang WaitingForCallback, N8nIntegrationService trigger webhook để LLM sinh test cases theo contract và prompt config.

5. Callback lưu test case

n8n gọi endpoint from-ai, backend validate payload rồi SaveAiGeneratedTestCasesCommand lưu request, expectation, dependency và trace.

Luồng code và bảng dữ liệu

  • FE: GeneratingTestCasesPage gọi POST /api/test-suites/{suiteId}/generate-tests và poll /generation-status.
  • Controller: TestOrderController dispatch GenerateTestCasesCommand; command yêu cầu suite đã có approved proposal trong testgen.TestOrderProposals.
  • Generation job: job queue/callback được lưu trong testgen.TestGenerationJobs với trạng thái Queued, Triggering, WaitingForCallback, Completed hoặc Failed.
  • Background: TriggerTestGenerationConsumer gọi TestGenerationPayloadBuilder, đọc suite/order/SRS/API context rồi trigger n8n webhook.
  • Persist test cases: callback from-ai lưu vào testgen.TestCases, testgen.TestCaseRequests, testgen.TestCaseExpectations, testgen.TestCaseDependencies, testgen.TestCaseVariables và testgen.TestCaseRequirementLinks.
Bước Bảng được ghi/cập nhật Dữ liệu nằm trong bảng
Đọc suite và phạm vi generate testgen.TestSuites Suite chứa ProjectId, ApiSpecId, SelectedEndpointIds, EndpointBusinessContexts, GlobalBusinessRules, SrsDocumentId, trạng thái và approval để xác định scope.
Đọc order đã approve testgen.TestOrderProposals Command yêu cầu proposal đã approve/applied; dùng AppliedOrder, ProposalId, AiReasoning, ConsideredFactors để sinh case theo thứ tự hợp lý.
Tạo generation job testgen.TestGenerationJobs Job async lưu TestSuiteId, ProposalId, Status, TriggeredById, QueuedAt, TriggeredAt, CompletedAt, TestCasesGenerated, webhook/callback URL và lỗi nếu có.
Build payload API + SRS apidoc.*, testgen.Srs* Payload đọc endpoint, parameter, response, security từ schema apidoc và requirement/constraint/clarification từ testgen.SrsDocuments, testgen.SrsRequirements.
Lưu test case chính testgen.TestCases Mỗi test case lưu TestSuiteId, EndpointId, Name, Description, TestType, Priority, OrderIndex, Tags, PrimaryRequirementId, version và trạng thái enabled/deleted.
Lưu request của test case testgen.TestCaseRequests Request runtime-ready lưu method, URL/path template, headers, path params, query params, body type, body và timeout theo TestCaseId.
Lưu expected result testgen.TestCaseExpectations Expectation lưu ExpectedStatus, ResponseSchema, header/body/json path checks, max response time, ExpectationSource, requirement code và ExpectedProvenance.
Lưu dependency và biến testgen.TestCaseDependencies, testgen.TestCaseVariables Dependency xác định case nào cần chạy trước; variables mô tả giá trị sẽ extract từ response/header/status để case sau dùng lại.
Lưu trace requirement testgen.TestCaseRequirementLinks Mỗi link nối TestCaseId với SrsRequirementId, kèm TraceabilityScore và MappingRationale để report coverage có bằng chứng.

Vai trò trong luồng tổng thể

Workflow 6 là nơi tri thức được biến thành hành động kiểm thử. Các workflow trước chuẩn bị dữ liệu: environment, API contract, yêu cầu nghiệp vụ, mapping và dependency. Workflow 6 dùng toàn bộ dữ liệu đó để tạo test case cụ thể, có thể review và chạy thật.
Chất lượng test phụ thuộc trực tiếp vào context đầu vào. Nếu Swagger đúng, SRS rõ, merge tốt và parameter mapping đầy đủ, test sinh ra sẽ sát nghiệp vụ hơn, ít bị thiếu tham số hơn và dễ giải thích kết quả hơn ở Workflow 7.

Điều kiện để xem là sinh test case đúng

  • Mỗi test case có mục tiêu kiểm thử rõ ràng, không chỉ là request mẫu.
  • Request có đủ method, URL, headers, body và dữ liệu cần thiết.
  • Expected result đủ rõ để hệ thống biết pass/fail khi execute.
  • Dependency giữa các test case được ghi nhận nếu có dữ liệu cần dùng lại.
  • Test case có trace đến endpoint và requirement liên quan.
  • Nếu thiếu expectation, test có thể gọi API thành công nhưng không chứng minh được gì.
  • Nếu dữ liệu đầu vào không phù hợp, test fail vì test data chứ không phải vì API lỗi.
  • Nếu không có negative/boundary case, bộ test dễ chỉ kiểm tra trường hợp đẹp.
  • Nếu dependency sai, test suite có thể fail do thứ tự chạy thay vì lỗi nghiệp vụ.
  • Nếu không trace requirement, báo cáo coverage sẽ thiếu cơ sở thuyết phục.
Workflow 6 tạo ra bộ test case có thể review và execute. Đây là bước biến API contract, yêu cầu nghiệp vụ và cách truyền tham số thành các kịch bản kiểm thử cụ thể.
