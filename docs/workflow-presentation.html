Luồng Chạy Tổng Quan

1. EnvironmentNgười dùng cấu hình base URL, headers, token và variables.
2. SwaggerUpload spec, parse endpoints, request schema, response schema.
3. SRSUpload tài liệu yêu cầu, phân tích requirement và constraints.
4. MergeGhép API contract với business rules và traceability từ SRS.
5. ParametersTính cách truyền path, query, body, auth và biến phụ thuộc.
6. GenerateGọi LLM/n8n hoặc generator nội bộ để tạo test cases.
7. ExecuteChạy test, validate kết quả, ghi report và giải thích lỗi.

Trục chính của hệ thống là: Project → API Spec → SRS → Environment → Test Suite → Test Cases → Test Run → Validation/Reporting. Mỗi workflow tạo dữ liệu đầu vào cho workflow kế tiếp, nên thiếu dữ liệu ở bước trước sẽ làm giảm chất lượng sinh test hoặc làm test run thất bại.

Mô Tả Từng Workflow

Workflow 1 - Setup Environment Parameters

Được gọi khi: người dùng tạo hoặc chọn execution environment cho project.

Cách chạy: backend lưu base URL, headers, auth token và variables. Khi execute test, orchestrator resolve environment rồi merge với biến runtime đã extract từ các test trước.

Output: bộ tham số runtime dùng để gọi API thật.

Workflow 2 - Import and Analyze Swagger / OpenAPI

Được gọi khi: user upload file Swagger/OpenAPI hoặc chọn parse specification.

Cách chạy: parser đọc paths, methods, parameters, request body, response schema và security scheme; sau đó tạo endpoint metadata trong database.

Output: danh sách endpoint và API contract làm nguồn chính cho test generation.

Workflow 3 - Import and Analyze SRS

Được gọi khi: user upload SRS hoặc yêu cầu phân tích requirement.

Cách chạy: hệ thống trích xuất requirement, rule nghiệp vụ, constraint kiểm thử, ambiguity và confidence score. Nếu dùng LLM/n8n, payload có thể gửi SRS sang workflow phân tích.

Output: requirement có mã, mô tả, constraints và dữ liệu traceability.

Workflow 4 - Merge Swagger with SRS

Được gọi khi: cần tạo context cho LLM hoặc chuẩn bị test suite dựa trên cả API contract và business rules.

Cách chạy: endpoint từ Swagger được ghép với requirement/constraint từ SRS. Bước này xác định endpoint nào cover requirement nào và rule nào phải trở thành expectation.

Output: prompt/context hợp nhất cho generation và traceability link.

Workflow 5 - Explain How to Pass Parameters

Được gọi khi: hệ thống cần biết test case sau lấy dữ liệu từ test case trước như thế nào.

Cách chạy: dependency analyzer so khớp response fields với path/query/body parameters, resolve biến từ environment và variable bag, rồi mô tả cách truyền dữ liệu qua chuỗi test.

Output: mapping parameter, extracted variables và thứ tự chạy an toàn hơn.

Workflow 6 - Generate Test Cases

Được gọi khi: user yêu cầu sinh happy path, boundary, negative hoặc LLM suggestions.

Cách chạy: backend build payload từ suite scope, endpoints, approved order, SRS và environment context; sau đó gọi LLM/n8n hoặc fallback generator nội bộ, normalize request/expectation và lưu test cases.

Output: test cases có request, expected status, assertions, dependency và covered requirement IDs.

Workflow 7 - Execute Test Cases, AI Explain Result / Failure

Được gọi khi: user chạy test suite hoặc mở kết quả test run.

Cách chạy: orchestrator load test cases theo approved order, resolve variables, gọi API thật, validate status/schema/assertions, lưu pass/fail/skip. Với case fail, hệ thống có thể gọi LLM để giải thích nguyên nhân.

Output: test run report, execution logs, failure reason và đề xuất sửa lỗi.
