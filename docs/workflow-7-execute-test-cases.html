Seven Core Workflow
Workflow 7

Execute Test Cases, AI Explain Result / Failure

Chạy test case trên API thật, kiểm tra kết quả và giải thích nguyên nhân pass/fail

Workflow 7 là bước đưa bộ test case đã sinh vào thực thi. Hệ thống chọn environment, resolve dữ liệu cần truyền, gọi API thật theo đúng thứ tự, kiểm tra response với expectation, sau đó tạo report. Nếu test fail, hệ thống giải thích lỗi theo ngữ cảnh để người dùng hiểu vấn đề nằm ở đâu.

Execute
Explain Runtime variables, API calls, assertions, report và failure explanation.

Overview nhanh

Mục tiêu

Chạy test case trên môi trường đã chọn, xác định pass/fail/skip và tạo báo cáo có giải thích rõ ràng.

Đầu vào

Approved test cases từ Workflow 6, environment từ Workflow 1, parameter mapping từ Workflow 5 và expected results.

Đầu ra

Test run report gồm trạng thái từng case, request/response log, assertion result, extracted variables và AI failure explanation.

Các bước chạy chuẩn của Workflow 7

Chọn test suite cần chạy

Người dùng chọn bộ test case đã review/approve và chọn environment muốn execute như Local, Staging hoặc Production-like.

Chuẩn bị runtime context

Hệ thống nạp base URL, auth, headers, variables, secrets và các tùy chọn chạy như timeout hoặc retry.

Sắp xếp thứ tự chạy

Test case được chạy theo dependency: case tạo dữ liệu hoặc lấy token sẽ chạy trước case cần dùng dữ liệu đó.

Resolve tham số request

Trước mỗi request, hệ thống điền path, query, header và body bằng giá trị từ environment hoặc kết quả các bước trước.

Gọi API thật

Request được gửi đến server thật. Hệ thống ghi lại thời gian chạy, status code, response body và lỗi kết nối nếu có.

Validate kết quả

Response được so với expected status, schema, assertions và business expectation đã gắn trong test case.

Extract dữ liệu cho bước sau

Nếu response có id, token hoặc giá trị cần dùng tiếp, hệ thống lưu vào runtime variables để request sau có thể dùng.

Tạo report và giải thích lỗi

Khi hoàn tất, hệ thống tạo báo cáo pass/fail/skip. Với case fail, AI giải thích nguyên nhân có thể xảy ra và hướng kiểm tra tiếp.

Bên trong workflow hoạt động ra sao?

1. FE start test run

Trang execute gọi POST test-runs với suiteId, environmentId, selected test cases và retry/validation policy nếu có.

2. Backend tạo run

StartTestRunCommandHandler kiểm tra owner, suite Ready, case hợp lệ, quota, environment rồi tạo TestRun trạng thái Pending.

3. Orchestrator chạy case

TestExecutionOrchestrator load ordered test cases, resolve environment, thay variables, xử lý dependency rồi gọi HTTP thật.

4. Validate và retry

Kết quả API được kiểm status, schema, headers, body, json path và response time; case fail có thể retry hoặc làm case phụ thuộc bị skip.

5. Collect report

ResultCollector lưu pass/fail/skip, attempts, extracted variables, failure reasons và dữ liệu chi tiết để FE hiển thị report.

Luồng code và bảng dữ liệu

  • FE: GenerationRunExecutePage hoặc TestRunsPage gọi POST /api/test-suites/{suiteId}/test-runs với environmentId, selected cases và retry policy.
  • Controller: TestRunsController dispatch StartTestRunCommand; command kiểm tra suite, owner, case IDs, quota và environment.
  • Persist run: StartTestRunCommandHandler tạo một bản ghi trong testexecution.TestRuns với RunNumber, EnvironmentId, Status và RedisKey.
  • Execute: TestExecutionOrchestrator đọc test cases từ testgen.TestCases cùng request, expectation, dependency và variables liên quan.
  • Persist result: TestResultCollector ghi pass/fail/skip vào testexecution.TestCaseResults; summary count và trạng thái cuối cập nhật lại testexecution.TestRuns.
Bước Bảng được ghi/cập nhật Dữ liệu nằm trong bảng
Chọn environment testexecution.ExecutionEnvironments Runner đọc BaseUrl, Variables, Headers, AuthConfig theo EnvironmentId hoặc default để dựng runtime context.
Tạo test run testexecution.TestRuns Start command tạo run với TestSuiteId, EnvironmentId, TriggeredById, RunNumber, Status, TotalTests, RedisKey, StartedAt.
Đọc test case để chạy testgen.TestCases Lấy case enabled theo suite, gồm EndpointId, Name, TestType, Priority, OrderIndex, CustomOrderIndex và requirement chính nếu có.
Resolve request testgen.TestCaseRequests, testgen.TestCaseVariables Request cung cấp URL, params, headers, body và timeout; variable definitions cho biết phải extract field nào sau mỗi response để thay placeholder ở case sau.
Validate expectation testgen.TestCaseExpectations Runner kiểm ExpectedStatus, response schema, header checks, body contains/not contains, json path checks, response time và provenance nghiệp vụ.
Tôn trọng dependency testgen.TestCaseDependencies Quan hệ TestCaseId - DependsOnTestCaseId quyết định thứ tự chạy; nếu dependency fail, case phụ thuộc có thể bị skip kèm lý do.
Lưu kết quả từng case testexecution.TestCaseResults Mỗi result lưu TestRunId, TestCaseId, EndpointId, Status, HttpStatusCode, DurationMs, ResolvedUrl, request/response headers, body preview, failure reasons, extracted variables và validation flags.
Cập nhật summary run testexecution.TestRuns Sau khi chạy xong, run cập nhật Status, CompletedAt, PassedCount, FailedCount, SkippedCount, DurationMs, ResultsExpireAt.
Report lạnh khi Redis hết hạn testexecution.TestCaseResults Result được lưu ở PostgreSQL như cold storage để FE hoặc AI failure explanation vẫn đọc được bằng chứng sau khi cache chi tiết trong Redis hết hạn.

Vai trò trong luồng tổng thể

Workflow 7 là bước kiểm chứng thực tế. Sáu workflow trước chuẩn bị context, mapping và test case. Workflow 7 đưa mọi thứ vào môi trường thật để biết API đang hoạt động đúng, sai hay thiếu điều kiện chạy.
AI explanation giúp report dễ hành động hơn. Thay vì chỉ thấy fail, người dùng có thêm phần giải thích: lỗi có thể do auth, test data, dependency, schema mismatch, business rule không đạt hoặc server trả response khác kỳ vọng.

Điều kiện để xem là execute đúng

  • Test suite chạy đúng environment và đúng base URL đã chọn.
  • Auth, headers, variables và secrets được resolve trước khi gửi request.
  • Dependency giữa các test case được tôn trọng trong thứ tự chạy.
  • Kết quả pass/fail dựa trên expectation rõ ràng, không chỉ dựa vào API có trả response hay không.
  • Report ghi lại đủ request, response, assertion và lý do fail để người dùng review.
  • Nếu chọn sai environment, toàn bộ kết quả có thể không phản ánh hệ thống cần kiểm thử.
  • Nếu token hết hạn hoặc secret sai, nhiều case sẽ fail vì setup thay vì lỗi API.
  • Nếu dependency không đúng, case sau có thể fail vì thiếu dữ liệu từ case trước.
  • Nếu expectation quá mơ hồ, hệ thống khó phân biệt pass thật với response chỉ có vẻ hợp lệ.
  • Nếu log không đủ, AI explanation và người dùng đều khó truy ra nguyên nhân thật.
Workflow 7 đóng vòng kiểm thử: từ test case đã sinh, hệ thống chạy thật, kiểm tra thật, ghi nhận bằng chứng và giải thích kết quả để người dùng biết bước tiếp theo nên xử lý ở đâu.
