Thuật toán sắp thứ tự endpoint

Dependency-aware topological sorting

Thuật toán này biến danh sách endpoint và các cạnh phụ thuộc thành thứ tự chạy test hợp lý: endpoint tạo dữ liệu, đăng ký, đăng nhập hoặc có nhiều endpoint khác cần dùng sẽ được ưu tiên chạy trước.

consumer later

Nằm ở đâu và được gọi khi nào?

Implementation

DependencyAwareTopologicalSorter implement IDependencyAwareTopologicalSorter trong thư mục Algorithms.

Điểm gọi chính

ApiTestOrderAlgorithm.BuildProposalOrder gom dependency edges rồi gọi _topologicalSorter.Sort(...).

Vai trò workflow

Nó tạo OrderIndex cho proposal, từ đó Workflow 6 sinh test case theo đúng thứ tự dữ liệu và Workflow 7 execute ít bị fail do thiếu setup.

Đầu vào và đầu ra

Đầu vào

  • SortableOperation: OperationId, HTTP method, path, cờ IsAuthRelated.
  • DependencyEdge: cạnh có hướng, nghĩa là SourceOperationId phụ thuộc vào TargetOperationId.
  • Cạnh confidence thấp hơn 0.5 vẫn được ghi metadata nhưng không ép thứ tự.

Đầu ra

  • SortedOperationResult gồm OrderIndex, Dependencies, DependencyEdges.
  • FanOut cho biết endpoint này được bao nhiêu endpoint khác cần.
  • ReasonCodes giải thích vì sao endpoint được xếp ở vị trí đó.

Luồng thuật toán

Khử trùng endpoint

Group theo OperationId để mỗi endpoint chỉ xuất hiện một lần trong graph.

Dựng graph phụ thuộc

Tạo dependenciesMap và dependentsMap. Mỗi cạnh source - target nghĩa là source cần target chạy trước.

Tính in-degree và fan-out

inDegree là số dependency còn thiếu. fanOut là số endpoint phụ thuộc vào node hiện tại.

Chọn node sẵn sàng

Node có inDegree = 0 được đưa vào tập available để có thể chạy tiếp.

Xếp hạng ứng viên

Ưu tiên auth trước, fan-out cao trước, in-degree thấp, method weight, path alphabet và cuối cùng là Guid để deterministic.

Ghi kết quả

Endpoint được chọn nhận OrderIndex, reason code và danh sách dependency metadata.

Giảm in-degree

Sau khi một producer được xếp, các consumer của nó được giảm in-degree; nếu về 0 thì vào hàng đợi.

Phá cycle nếu cần

Nếu không có node sẵn sàng, chọn node có in-degree thấp và fan-out cao để phá vòng, đánh dấu CYCLE_BREAK_FALLBACK.

Cách được dùng trong workflow sinh test

Trong Workflow 4/6

Thứ tự proposal giúp hệ thống biết endpoint nào tạo dữ liệu trước. Ví dụ: POST /users hoặc endpoint login thường cần đứng trước GET /users/{id} hoặc endpoint cần bearer token.

Khi callback lưu test case

OrderIndex và dependency chain được dùng để lưu test case có thứ tự, giúp runtime resolver lấy biến từ case trước cho case sau.

Khi debug cần nhìn gì?

  • Nếu thứ tự sai, kiểm tra DependencyEdge.SourceOperationId và TargetOperationId có bị đảo chiều không.
  • Nếu cạnh xuất hiện trong metadata nhưng không ép thứ tự, kiểm tra Confidence có thấp hơn 0.5 không.
  • Nếu có cycle, xem IsCycleBreak và reason code để biết node nào bị chọn làm điểm phá vòng.
