Seven Core Workflow
Workflow 3

Import and Analyze SRS

Đưa tài liệu yêu cầu vào hệ thống và biến nội dung nghiệp vụ thành ngữ cảnh kiểm thử

Workflow 3 là bước hệ thống đọc SRS để hiểu nghiệp vụ muốn gì, rule nào cần kiểm tra, điều kiện nào phải được thỏa mãn và những yêu cầu nào cần được trace khi sinh test case. Nếu Workflow 2 mô tả API về mặt kỹ thuật, thì Workflow 3 bổ sung phần ý nghĩa nghiệp vụ phía sau API đó.

SRS
Analysis Requirements, business rules, constraints, scenarios và traceability.

Overview nhanh

Mục tiêu

Chuyển tài liệu SRS từ dạng văn bản dài thành danh sách yêu cầu, rule, constraint và scenario có thể dùng để sinh test.

Đầu vào

File SRS, mô tả nghiệp vụ, use case, acceptance criteria, business constraints và project đang chọn.

Đầu ra

Bộ requirement đã phân tích gồm mã yêu cầu, mô tả, rule liên quan, độ ưu tiên, rủi ro và thông tin traceability.

Các bước chạy chuẩn của Workflow 3

Upload hoặc chọn SRS

Người dùng đưa tài liệu SRS vào project, có thể là file yêu cầu, tài liệu đặc tả hoặc nội dung mô tả nghiệp vụ.

Kiểm tra tài liệu

Hệ thống kiểm tra tài liệu có đọc được không, có nội dung đủ rõ không và có đúng phạm vi project đang phân tích không.

Tách phần nội dung chính

Các đoạn liên quan đến chức năng, rule nghiệp vụ, luồng người dùng, dữ liệu và điều kiện kiểm thử được tách ra để xử lý.

Nhận diện requirement

Hệ thống gom các yêu cầu thành từng requirement rõ ràng, tránh để một đoạn văn dài chứa quá nhiều ý kiểm thử lẫn vào nhau.

Phân tích rule và constraint

Các điều kiện như bắt buộc, giới hạn dữ liệu, phân quyền, trạng thái hợp lệ và ngoại lệ nghiệp vụ được ghi nhận.

Xác định scenario kiểm thử

Từ requirement, hệ thống suy ra các hướng kiểm thử như happy path, negative case, boundary case và exception case.

Đánh dấu điểm chưa rõ

Nếu tài liệu mơ hồ, thiếu dữ liệu hoặc có rule xung đột, hệ thống ghi nhận để người dùng review trước khi sinh test.

Lưu kết quả phân tích

Kết quả SRS analysis được lưu lại để Workflow 4 merge với Swagger và Workflow 6 dùng làm ngữ cảnh sinh test case.

Bên trong workflow hoạt động ra sao?

1. FE tạo SRS document

SrsDocumentsPage gửi title, source type, raw content hoặc storage file id để backend tạo SrsDocument trong project.

2. Trigger analysis job

Khi người dùng bấm Analyze, backend tạo SrsAnalysisJob, chuyển document sang Processing và lấy nội dung text cần phân tích.

3. Build payload cho n8n

Command gom raw content, project context, endpoint references và callback URL rồi gọi webhook analyze-srs qua N8nIntegrationService.

4. Nhận requirement kết quả

Nếu n8n trả kết quả ngay hoặc callback về sau, ProcessSrsAnalysisCallbackCommand lưu requirements, constraints và clarification questions.

5. Tạo business context

SRS requirements sau khi phân tích được dùng để link endpoint, build traceability và đưa rule nghiệp vụ vào test generation.

Luồng code và bảng dữ liệu

  • FE: SrsDocumentsPage dùng srsService gọi /api/projects/{projectId}/srs-documents để tạo document và gọi /analyze để phân tích.
  • Controller: SrsDocumentsController dispatch CreateSrsDocumentCommand, TriggerSrsAnalysisCommand và các query xem job/status.
  • Persist document: nội dung SRS, source type, linked suite và analysis status lưu trong testgen.SrsDocuments.
  • Analysis job: TriggerSrsAnalysisCommand tạo job trong testgen.SrsAnalysisJobs, build payload rồi gọi n8n webhook analyze-srs.
  • Persist result: callback hoặc response đồng bộ được ProcessSrsAnalysisCallbackCommand lưu vào testgen.SrsRequirements và testgen.SrsRequirementClarifications.
Bước Bảng được ghi/cập nhật Dữ liệu nằm trong bảng
Tạo SRS document testgen.SrsDocuments Một dòng tài liệu theo project, gồm ProjectId, TestSuiteId nếu đã link suite, Title, SourceType, RawContent, StorageFileId, ParsedMarkdown, AnalysisStatus, CreatedById.
Upload file SRS classifiedads_storage.FileEntries Nếu nguồn là file upload, Storage module giữ file gốc; testgen.SrsDocuments.StorageFileId trỏ tới file này để pipeline trích xuất nội dung.
Trigger analysis testgen.SrsAnalysisJobs Job async lưu SrsDocumentId, Status, TriggeredById, QueuedAt, TriggeredAt, CompletedAt, RequirementsExtracted, RetryCount, JobType và lỗi nếu có.
Đánh dấu document đang xử lý testgen.SrsDocuments AnalysisStatus chuyển qua Processing, sau callback thì thành Completed hoặc Failed; AnalyzedAt được set khi hoàn tất.
Lưu requirement đã tách testgen.SrsRequirements Mỗi requirement thành một dòng theo SrsDocumentId, lưu RequirementCode, Title, Description, RequirementType, TestableConstraints, Assumptions, Ambiguities, ConfidenceScore, EndpointId, MappedEndpointPath.
Lưu câu hỏi làm rõ testgen.SrsRequirementClarifications Các điểm mơ hồ được lưu theo SrsRequirementId, gồm AmbiguitySource, Question, SuggestedOptions, UserAnswer, IsAnswered, DisplayOrder và IsCritical.

Vai trò trong luồng tổng thể

Workflow 3 là nguồn sự thật nghiệp vụ. Swagger cho biết API nhận và trả dữ liệu gì, nhưng SRS cho biết vì sao API tồn tại, hành vi nào được xem là đúng và trường hợp nào phải bị từ chối. Nhờ đó test case không chỉ đúng format, mà còn đúng mục tiêu nghiệp vụ.
Kết quả SRS analysis giúp tăng chất lượng test generation. Khi hệ thống biết rule, constraint và scenario, Workflow 6 có thể sinh test case sát thực tế hơn: không chỉ gọi endpoint, mà còn kiểm tra đúng điều kiện thành công, thất bại và biên dữ liệu.

Điều kiện để xem là phân tích SRS đúng

  • Các requirement chính được nhận diện đầy đủ và có mô tả dễ hiểu.
  • Business rules và constraints quan trọng được tách riêng, không bị lẫn vào mô tả chung.
  • Các scenario kiểm thử có thể suy ra được từ từng requirement.
  • Điểm mơ hồ, thiếu dữ liệu hoặc xung đột trong tài liệu được đánh dấu để review.
  • Kết quả phân tích có thể liên kết với endpoint ở Workflow 4.
  • Nếu SRS quá chung chung, test sinh ra có thể chỉ kiểm tra bề mặt, không bắt được rule thật.
  • Nếu requirement bị tách sai, một test case có thể cover nhầm mục tiêu nghiệp vụ.
  • Nếu bỏ sót constraint, negative case và boundary case sẽ yếu hơn.
  • Nếu tài liệu có rule xung đột mà không đánh dấu, kết quả generate test có thể không nhất quán.
  • Nếu không lưu traceability, sau này khó chứng minh test case đang cover requirement nào.
Workflow 3 biến SRS thành requirement context. Context này giúp hệ thống hiểu ý nghĩa nghiệp vụ, nối yêu cầu với API và sinh test case có khả năng kiểm tra đúng hành vi mong đợi.
