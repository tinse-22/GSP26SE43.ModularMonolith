Seven Core Workflow
Workflow 4

Merge Swagger with SRS

Ghép API contract với yêu cầu nghiệp vụ để tạo bản đồ kiểm thử có trace rõ ràng

Workflow 4 là bước nối hai nguồn thông tin quan trọng: Swagger/OpenAPI cho biết API hoạt động ra sao, còn SRS cho biết nghiệp vụ mong đợi điều gì. Sau khi merge, hệ thống biết endpoint nào phục vụ requirement nào, rule nào cần kiểm tra, và test case sinh ra sẽ cover mục tiêu nào.

API + SRS
Merge Endpoint mapping, business rules, coverage, traceability và generation context.

Overview nhanh

Mục tiêu

Liên kết endpoint từ Swagger với requirement từ SRS để tạo context thống nhất cho việc sinh test case.

Đầu vào

API catalog từ Workflow 2, requirement context từ Workflow 3, project scope và thông tin environment từ Workflow 1.

Đầu ra

Bản merge gồm endpoint-requirement mapping, rule cần kiểm tra, coverage gap và traceability cho test generation.

Các bước chạy chuẩn của Workflow 4

Chọn bộ Swagger và SRS

Người dùng chọn API catalog đã import và bộ requirement đã phân tích để hệ thống biết hai nguồn nào cần ghép với nhau.

Đồng bộ phạm vi

Hệ thống kiểm tra version, module, feature hoặc project scope để tránh ghép nhầm API của bản này với SRS của bản khác.

So khớp endpoint với yêu cầu

Endpoint được gợi ý liên kết với requirement dựa trên tên chức năng, đường dẫn API, mô tả, dữ liệu vào ra và hành vi mong đợi.

Gắn business rule vào API

Các rule trong SRS được đưa vào endpoint liên quan để biết request nào cần kiểm tra điều kiện nghiệp vụ nào.

Phát hiện khoảng trống

Hệ thống chỉ ra requirement chưa có endpoint phù hợp hoặc endpoint chưa thấy requirement tương ứng để người dùng review.

Tạo traceability

Mỗi liên kết được ghi lại để sau này biết một test case được sinh ra từ endpoint nào và đang cover requirement nào.

Chuẩn bị context sinh test

Thông tin API, rule, scenario và constraint được gom thành một bộ ngữ cảnh rõ ràng cho Workflow 6.

Lưu kết quả merge

Kết quả được lưu để dùng lại khi giải thích tham số, generate test case, execute test và xem báo cáo coverage.

Bên trong workflow hoạt động ra sao?

1. Load API và SRS context

Backend lấy endpoint metadata từ ApiDocumentation và lấy SRS document, requirements, traceability links từ TestGeneration.

2. So khớp endpoint với rule

Hệ thống đối chiếu path, method, operation, parameter, response với requirement title, constraint và endpoint mapping nếu có.

3. Tạo traceability link

Requirement được gắn với test suite hoặc endpoint để biết test case sau này đang cover rule nghiệp vụ nào.

4. Nén thành prompt context

TestGenerationPayloadBuilder gom business context theo endpoint, global rules và requirement count để payload không quá dài.

5. Cấp dữ liệu cho generation

Context hợp nhất được đưa vào Workflow 6 để LLM sinh test case có endpoint thật, expectation rõ và requirement trace.

Luồng code và bảng dữ liệu

  • Nguồn API: backend đọc endpoint contract từ apidoc.ApiEndpoints, apidoc.EndpointParameters và apidoc.EndpointResponses.
  • Nguồn SRS: backend đọc business rules từ testgen.SrsDocuments và testgen.SrsRequirements.
  • Link suite: SRS document có thể được gắn với test suite qua testgen.TestSuites.SrsDocumentId hoặc testgen.SrsDocuments.TestSuiteId.
  • Traceability: khi link requirement với test case, CreateTraceabilityLinkCommand ghi vào testgen.TestCaseRequirementLinks.
  • Generation context: TestGenerationPayloadBuilder không tạo bảng merge riêng; nó đọc các bảng trên, nén thành payload và gửi sang Workflow 6.
Bước Bảng được ghi/cập nhật Dữ liệu nằm trong bảng
Đọc API catalog apidoc.ApiSpecifications, apidoc.ApiEndpoints Lấy spec/version active và từng endpoint theo ApiSpecId, gồm method, path, operation id, summary, description, tags và trạng thái deprecated.
Đọc input/output API apidoc.EndpointParameters, apidoc.EndpointResponses Lấy parameter theo EndpointId để biết dữ liệu request cần truyền, và response schema/status code để biết kết quả endpoint có thể trả về.
Đọc context nghiệp vụ testgen.SrsDocuments, testgen.SrsRequirements Lấy SRS document đã phân tích, requirement code, title, description, constraints, assumptions, ambiguities, confidence và endpoint mapping nếu đã có.
Link SRS với test suite testgen.TestSuites, testgen.SrsDocuments TestSuites.SrsDocumentId hoặc SrsDocuments.TestSuiteId giữ quan hệ suite nào đang dùng bộ requirement nào để generate test.
Lưu business context theo suite testgen.TestSuites ApiSpecId, SelectedEndpointIds, EndpointBusinessContexts và GlobalBusinessRules giữ phạm vi endpoint và rule nghiệp vụ được đưa vào prompt generation.
Lưu traceability sau khi có test case testgen.TestCaseRequirementLinks Mỗi link lưu TestCaseId, SrsRequirementId, TraceabilityScore và MappingRationale; đây là bằng chứng test case cover requirement nào.
Tạo payload merge cho Workflow 6 Không tạo bảng merge riêng TestGenerationPayloadBuilder đọc các bảng trên rồi nén thành payload runtime cho LLM; dữ liệu merge chỉ được materialize thành test case/trace khi Workflow 6 lưu kết quả.

Vai trò trong luồng tổng thể

Workflow 4 là cầu nối giữa kỹ thuật và nghiệp vụ. Nếu chỉ có Swagger, test case dễ dừng ở mức gọi API đúng format. Nếu chỉ có SRS, hệ thống lại chưa biết endpoint thật nào để gọi. Bước merge giúp hai phần này gặp nhau để test case vừa gọi đúng API, vừa kiểm tra đúng rule.
Kết quả merge quyết định chất lượng sinh test. Khi endpoint được nối đúng với requirement, Workflow 6 có thể tạo test case có trace rõ ràng, còn Workflow 7 có thể báo pass/fail theo cả API behavior lẫn coverage nghiệp vụ.

Điều kiện để xem là merge đúng

  • Các requirement quan trọng có endpoint liên quan hoặc được đánh dấu là chưa có API phù hợp.
  • Mỗi mapping có lý do rõ ràng, không chỉ ghép theo tên giống nhau.
  • Business rules và constraints được gắn vào đúng endpoint cần kiểm tra.
  • Các điểm mơ hồ được đánh dấu để review thay vì âm thầm dùng làm dữ liệu sinh test.
  • Traceability đủ rõ để biết test case sau này cover requirement nào.
  • Nếu ghép sai endpoint với requirement, test case có thể pass nhưng vẫn không kiểm tra đúng nghiệp vụ.
  • Nếu bỏ sót rule từ SRS, negative case và boundary case sẽ thiếu chiều sâu.
  • Nếu không phát hiện coverage gap, team có thể tưởng đã test đủ trong khi vẫn còn yêu cầu chưa được cover.
  • Nếu mapping confidence thấp mà không review, kết quả generate test dễ bị lệch ngữ cảnh.
  • Nếu không lưu traceability, báo cáo cuối cùng khó chứng minh test đang bảo vệ yêu cầu nào.
Workflow 4 tạo context hợp nhất giữa API contract và yêu cầu nghiệp vụ. Đây là nền tảng để giải thích tham số, sinh test case có ý nghĩa và đo coverage theo requirement.
