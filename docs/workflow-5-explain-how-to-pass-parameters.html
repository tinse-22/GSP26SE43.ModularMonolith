Workflow 5

Explain How to Pass Parameters

Giải thích cách truyền tham số để mỗi request biết lấy dữ liệu từ đâu và đặt vào đâu

Workflow 5 là bước làm rõ cách dữ liệu di chuyển trong test suite. Hệ thống không chỉ biết endpoint cần tham số gì, mà còn phải hiểu tham số đó đến từ environment, từ người dùng nhập sẵn, hay từ kết quả của test case trước đó. Nhờ vậy, chuỗi test có thể chạy liền mạch thay vì từng request bị tách rời.

Parameter
Flow Path, query, header, body, environment variables và extracted values.

Overview nhanh

Mục tiêu

Giải thích rõ mỗi tham số cần truyền vào request sẽ lấy từ nguồn nào, truyền vào vị trí nào và phụ thuộc vào bước nào.

Đầu vào

Environment từ Workflow 1, API catalog từ Workflow 2, context merge từ Workflow 4 và danh sách test case dự kiến.

Đầu ra

Bản parameter mapping gồm nguồn dữ liệu, vị trí truyền, biến cần extract, thứ tự chạy và phần giải thích dễ hiểu cho người dùng.

Các bước chạy chuẩn của Workflow 5

Chọn endpoint hoặc test suite

Người dùng chọn endpoint cần giải thích tham số hoặc chọn cả test suite để xem dữ liệu truyền qua nhiều bước.

Đọc yêu cầu tham số

Hệ thống xem endpoint cần path, query, header, body field nào và tham số nào bắt buộc phải có.

Tìm nguồn dữ liệu phù hợp

Mỗi tham số được so với environment variables, dữ liệu test mẫu và response của các request trước để tìm nguồn hợp lý.

Xác định vị trí truyền

Hệ thống chỉ rõ dữ liệu sẽ đi vào URL path, query string, request header hay request body.

Nhận diện dependency

Nếu request sau cần dữ liệu từ request trước, hệ thống ghi nhận quan hệ phụ thuộc để biết bước nào phải chạy trước.

Đề xuất biến cần extract

Khi response trả về id, token hoặc mã giao dịch, hệ thống đề xuất lưu giá trị đó thành biến dùng cho các bước sau.

Giải thích bằng ngôn ngữ thường

Người dùng nhìn thấy lời giải thích: lấy giá trị nào, từ đâu, truyền vào đâu, và vì sao tham số đó cần thiết.

Lưu mapping cho execution

Kết quả mapping được lưu để Workflow 6 sinh test đúng hơn và Workflow 7 resolve dữ liệu khi chạy thật.

Bên trong workflow hoạt động ra sao?

1. Đọc contract và order

Backend lấy endpoint parameters từ API catalog và thứ tự endpoint đã approve trong test order proposal.

2. Tìm nguồn giá trị

Parameter được so với environment variables, request mẫu, response schema và output của các test case chạy trước.

3. Gắn dependency

Nếu case sau cần id/token/object từ case trước, hệ thống ghi dependency để runner không gọi sai thứ tự.

4. Resolve placeholder lúc chạy

Trong execution, VariableExtractor lấy dữ liệu từ response và VariableResolver thay placeholder vào path, query, header hoặc body.

5. Xử lý khi thiếu dữ liệu

Nếu dependency fail hoặc biến không resolve được, runner đánh dấu skip/fail có lý do thay vì gọi API với tham số rỗng.

Luồng code và bảng dữ liệu

  • Đọc contract: hệ thống lấy parameter/schema từ apidoc.EndpointParameters, apidoc.EndpointResponses và endpoint metadata service.
  • Đọc thứ tự: ProposeApiTestOrderCommandHandler và order gate dùng testgen.TestOrderProposals để biết endpoint nào chạy trước.
  • Persist dependency: khi test case được sinh/lưu, dependency giữa các case nằm trong testgen.TestCaseDependencies.
  • Persist request variables: request, placeholder và biến cần extract nằm ở testgen.TestCaseRequests và testgen.TestCaseVariables.
  • Runtime resolve: lúc chạy, VariableExtractor và VariableResolver dùng dữ liệu từ response hiện tại; giá trị runtime không có bảng riêng mà được lưu cùng execution result trong Workflow 7.
Bước Bảng được ghi/cập nhật Dữ liệu nằm trong bảng
Đọc parameter contract apidoc.EndpointParameters Lấy từng tham số theo EndpointId, gồm Name, Location, DataType, Format, ContentType, IsRequired, schema và examples để biết phải truyền gì.
Đọc response để gợi ý extract apidoc.EndpointResponses Lấy StatusCode, Schema, Examples và Headers để suy ra field nào có thể lấy làm biến cho request sau.
Đọc biến môi trường testexecution.ExecutionEnvironments Variables, Headers, AuthConfig và BaseUrl là nguồn dữ liệu nền cho placeholder, auth header và URL runtime.
Đọc thứ tự chạy đã duyệt testgen.TestOrderProposals AppliedOrder hoặc UserModifiedOrder cho biết endpoint/test case nào chạy trước, cùng AiReasoning và ConsideredFactors giải thích lý do order.
Lưu request mapping testgen.TestCaseRequests Mỗi test case có một request lưu HttpMethod, Url, Headers, PathParams, QueryParams, BodyType, Body và Timeout; placeholder nằm trong các JSON/text này.
Lưu biến cần extract testgen.TestCaseVariables Mỗi biến theo TestCaseId lưu VariableName, ExtractFrom, JsonPath, HeaderName, Regex và DefaultValue để runner biết lấy giá trị từ đâu.
Lưu dependency giữa case testgen.TestCaseDependencies Quan hệ TestCaseId phụ thuộc DependsOnTestCaseId giúp runner chạy đúng thứ tự và skip case sau nếu dữ liệu từ case trước bị fail.
Lưu giá trị runtime sau khi chạy testexecution.TestCaseResults Khi Workflow 7 execute, biến đã extract được ghi snapshot vào ExtractedVariables; URL/header đã resolve và failure do thiếu biến nằm trong result của từng case.

Vai trò trong luồng tổng thể

Workflow 5 biến API rời rạc thành chuỗi test có logic. Nhiều API không thể chạy độc lập: muốn lấy chi tiết phải có id, muốn cập nhật phải có bản ghi, muốn xóa phải biết đối tượng nào cần xóa. Bước này giúp hệ thống hiểu dữ liệu được tạo ở đâu và dùng lại thế nào.
Đây là cầu nối giữa generation và execution. Workflow 6 cần mapping để sinh test case có dependency hợp lý, còn Workflow 7 cần mapping để lấy giá trị thật sau mỗi response và truyền tiếp cho request kế tiếp.

Điều kiện để xem là giải thích tham số đúng

  • Mỗi tham số bắt buộc đều có nguồn dữ liệu rõ ràng hoặc được đánh dấu là thiếu dữ liệu.
  • Vị trí truyền được xác định đúng: path, query, header hoặc body.
  • Biến lấy từ response trước có bước extract rõ ràng và tên biến dễ hiểu.
  • Thứ tự chạy phản ánh đúng dependency giữa các test case.
  • Người dùng có thể đọc giải thích và hiểu vì sao request cần tham số đó.
  • Nếu truyền sai vị trí, API có thể trả lỗi dù giá trị tham số vẫn đúng.
  • Nếu thiếu dependency, test sau có thể chạy trước khi dữ liệu cần thiết được tạo.
  • Nếu extract sai field từ response, các bước sau sẽ dùng nhầm id hoặc nhầm token.
  • Nếu dùng dữ liệu hard-code quá nhiều, test khó tái chạy trên môi trường khác.
  • Nếu không giải thích nguồn dữ liệu, người dùng khó biết lỗi nằm ở test data hay API.
Workflow 5 tạo bản đồ truyền tham số cho test suite. Nó giúp hệ thống biết dữ liệu đến từ đâu, đi qua request nào và được dùng lại thế nào khi sinh test hoặc chạy test thật.
