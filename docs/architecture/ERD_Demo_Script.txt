MODULE 1: IDENTITY (schema "identity") - 9 bảng
Chức năng: Quản lý xác thực và phân quyền
────────────────────────────────────────────────────────────────────────────────

Các bảng:
  - Users              : Thông tin tài khoản (email, password hash, lockout...)
  - Roles              : Vai trò (Admin, User)
  - UserRoles          : Bảng trung gian N-N giữa Users và Roles
  - UserClaims         : Quyền hạn gắn trực tiếp cho user
  - RoleClaims         : Quyền hạn gắn cho role (17 permission claims cho Admin)
  - UserLogins         : Đăng nhập bên ngoài (Auth0, Azure AD B2C)
  - UserTokens         : Token của user (refresh token, 2FA...)
  - UserProfiles       : Thông tin hiển thị (avatar, timezone) - tách khỏi auth
  - DataProtectionKeys : Key mã hóa của ASP.NET Data Protection

Mối quan hệ:
  Users ──1:N──> UserRoles <──N:1── Roles     (Quan hệ N-N qua bảng trung gian)
  Users ──1:1──> UserProfiles                  (Tách thông tin hiển thị)
  Users ──1:N──> UserClaims
  Users ──1:N──> UserLogins
  Users ──1:N──> UserTokens
  Roles ──1:N──> RoleClaims

Dữ liệu mẫu (Seed):
  - 2 Users: admin (tinvtse@gmail.com), user (user@example.com)
  - 2 Roles: Admin, User
  - 17 RoleClaims cho Admin role (các permission cụ thể)

Điểm nhấn khi demo:
  - Dùng ASP.NET Identity nhưng CUSTOM tên bảng (không dùng AspNet* mặc định)
  - Claims-based authorization: phân quyền linh hoạt theo permission,
    không phải chỉ theo role
  - Hỗ trợ đăng nhập bên thứ 3 (Auth0, Azure AD B2C)

MODULE 2: API DOCUMENTATION (schema "apidoc") - 10 bảng
Chức năng: Parse và lưu trữ cấu trúc API được import vào
────────────────────────────────────────────────────────────────────────────────

Các bảng:
  - Projects                : Dự án (Name, BaseUrl, OwnerId)
  - ApiSpecifications       : Phiên bản tài liệu API (file Swagger/OpenAPI)
  - ApiEndpoints            : Danh sách endpoint được parse ra
  - EndpointParameters      : Tham số của endpoint (query, path, header, cookie)
  - EndpointResponses       : Response definition theo HTTP status code
  - EndpointSecurityReqs    : Yêu cầu bảo mật của endpoint
  - SecuritySchemes         : Định nghĩa scheme bảo mật (JWT, API Key, OAuth2)
  - AuditLogEntries         : Nhật ký thao tác trong module
  - OutboxMessages          : Event chưa publish
  - ArchivedOutboxMessages  : Event đã publish (lưu trữ)
EndpointParameters - Lấy ví dụ endpoint GET /api/v1/products/{productId}?lang=vi&fields=name,price, mỗi tham số là 1 row: productId (Path, UUID, required), lang (Query, optional, default "en"), fields (Query, optional).

EndpointResponses - 3 rows cho 3 trường hợp: 200 (thành công, kèm JSON Schema đầy đủ), 404 (không tìm thấy), 500 (lỗi server). Mỗi row có schema jsonb để TestGen tự động assert.

EndpointSecurityReqs - 3 ví dụ: endpoint cần Bearer token, endpoint cần OAuth2 với scopes cụ thể, và endpoint public (không có row = ai cũng gọi được).

SecuritySchemes - 3 loại scheme: JWT Bearer (Authorization: Bearer eyJ...), API Key (X-API-Key: sk-abc... truyền qua header), OAuth2 (kèm cấu hình flows, tokenUrl, scopes trong cột Configuration jsonb).
Mối quan hệ:
  Projects ──1:N──> ApiSpecifications         (1 project có nhiều phiên bản spec)
  Projects ──1:1──> ApiSpecifications          (ActiveSpecId - spec đang active)
  ApiSpecifications ──1:N──> ApiEndpoints      (Parse spec ra endpoints)
  ApiSpecifications ──1:N──> SecuritySchemes   (Các scheme auth trong spec)
  ApiEndpoints ──1:N──> EndpointParameters     (Tham số của endpoint)
  ApiEndpoints ──1:N──> EndpointResponses      (Responses theo status code)
  ApiEndpoints ──1:N──> EndpointSecurityReqs   (Yêu cầu auth của endpoint)

Bảng này trả lời câu hỏi: "Khi gọi API này, client cần truyền những gì?"

Các cột quan trọng:
  - Name       : Tên tham số (vd: "productId", "lang", "x-api-key")
  - Location   : Tham số nằm Ở ĐÂU trong HTTP request
                  + Path   = trên URL path     (vd: /products/{productId})
                  + Query  = trên query string  (vd: /products?lang=vi)
                  + Header = trong HTTP header   (vd: x-api-key: abc123)
                  + Body   = trong request body  (vd: JSON payload)
                  + Cookie = trong cookie
  - DataType   : Kiểu dữ liệu (string, integer, boolean, array, object)
  - Format     : Định dạng chi tiết (uuid, date-time, email, int64)
  - IsRequired : Bắt buộc truyền hay không
  - DefaultValue : Giá trị mặc định nếu client không truyền
  - Schema (jsonb) : JSON Schema nếu tham số là object phức tạp
  - Examples (jsonb) : Giá trị ví dụ

Bảng này trả lời câu hỏi: "API này cần xác thực kiểu gì? Cần quyền gì?"

Các cột quan trọng:
  - SecurityType  : Loại xác thực (enum)
                    + Bearer       = JWT Token (phổ biến nhất)
                    + ApiKey       = API Key truyền qua header/query

                    (vd: "BearerAuth", "ApiKeyAuth")
  - Scopes (jsonb): Danh sách quyền OAuth2 cần có

Bảng này trả lời câu hỏi: "Hệ thống API hỗ trợ những cách xác thực nào?
                            Cấu hình chi tiết ra sao?"

Khác biệt với EndpointSecurityReqs:
  - SecuritySchemes = ĐỊNH NGHĨA chung (vd: "hệ thống dùng JWT, format Bearer")
  - EndpointSecurityReqs = ÁP DỤNG scheme nào cho endpoint cụ thể nào

  Quan hệ:  SecuritySchemes ──thuộc về──> ApiSpecification (cấp file spec)
            EndpointSecurityReqs ──thuộc về──> ApiEndpoint (cấp từng endpoint)
            EndpointSecurityReqs.SchemeName ──tham chiếu──> SecuritySchemes.Name

Các cột quan trọng:
  - Name           : Tên scheme (vd: "BearerAuth", "ApiKeyAuth")
  - Type           : Loại scheme (enum)
                     + Http          = HTTP Authentication (Bearer, Basic)
                     + ApiKey        = API Key
                     + OAuth2        = OAuth 2.0
                     + OpenIdConnect = OIDC
  - Scheme         : Sub-type của HTTP Auth (vd: "bearer", "basic")
  - BearerFormat   : Định dạng token (vd: "JWT")
  - In             : API Key nằm ở đâu (Header/Query/Cookie)
  - ParameterName  : Tên parameter của API Key (vd: "x-api-key", "api_key")
  - Configuration (jsonb) : Cấu hình phức tạp (OAuth2 flows, token URLs...)

TẠI SAO CẦN BẢNG NÀY?
  -> Khi TestGen sinh test cho 1 endpoint, nó đọc EndpointSecurityReqs
     để biết endpoint cần scheme nào (vd: "BearerAuth").
  -> Rồi đọc SecuritySchemes (theo SchemeName) để biết CHI TIẾT:
     + Bearer -> Cần gắn header "Authorization: Bearer <token>"
     + ApiKey -> Cần gắn header đúng tên (X-API-Key) đúng vị trí (Header)
     + OAuth2 -> Cần gọi tokenUrl lấy token trước, kèm đúng scopes
  -> Kết hợp với ExecutionEnvironments.AuthConfig (module TestExec)
     để lấy credentials thực tế (username/password test) cho từng môi trường.

================================================================================
               3 DESIGN PATTERNS QUAN TRỌNG (PHẢI NÊU KHI DEMO)
================================================================================

────────────────────────────────────────────────────────────────────────────────
PATTERN 1: OUTBOX PATTERN (Event-Driven Architecture)
────────────────────────────────────────────────────────────────────────────────

Vấn đề: Khi user thanh toán xong Subscription, cần cộng quota sử dụng.
         Nếu gửi event rồi hệ thống crash -> mất event -> dữ liệu không nhất quán.

Giải pháp: Mỗi module có bảng OutboxMessages + ArchivedOutboxMessages.
  1. Lưu dữ liệu nghiệp vụ + event vào OutboxMessages CÙNG 1 TRANSACTION
  2. Background worker đọc OutboxMessages và publish event
  3. Sau khi publish thành công -> chuyển vào ArchivedOutboxMessages

Kết quả: Đảm bảo EVENTUAL CONSISTENCY - không bao giờ mất event dù hệ thống
         crash giữa chừng.

Các module có Outbox: ApiDoc, TestGen, TestExec, TestReport, Subscription,
                      Storage, LlmAssistant (7/11 module)

────────────────────────────────────────────────────────────────────────────────
PATTERN 2: CROSS-MODULE REFERENCE (Không FK liên schema)
────────────────────────────────────────────────────────────────────────────────

Vấn đề: Trong Modular Monolith, nếu đặt FK constraint giữa các schema
         thì không thể tách module ra thành microservice sau này.

Giải pháp: Các module tham chiếu nhau qua GUID COLUMN + INDEX (không có FK).
  VD: testgen_TestSuites.ApiSpecId là Guid trỏ về apidoc_ApiSpecifications
      nhưng KHÔNG CÓ FK CONSTRAINT ở DB level.

Module boundary được enforce ở APPLICATION LAYER (code), không ở DB layer.
-> Dễ tách thành microservice sau này nếu cần scale.

────────────────────────────────────────────────────────────────────────────────
PATTERN 3: OPTIMISTIC CONCURRENCY CONTROL
────────────────────────────────────────────────────────────────────────────────

Vấn đề: 2 user cùng edit 1 test case cùng lúc -> user save sau sẽ ghi đè
         thay đổi của user save trước (Lost Update Problem).

Giải pháp: Mọi bảng chính đều có cột RowVersion (bytea/binary).
  1. Khi đọc record, đọc luôn RowVersion
  2. Khi update, EF Core kiểm tra RowVersion có đổi không
  3. Nếu RowVersion đã thay đổi (có người khác edit trước) -> throw exception
  4. Application xử lý conflict (thông báo user, merge, retry...)

Kết quả: Không cần LOCK database, vẫn đảm bảo data integrity.

           GỢI Ý CÂU HỎI THẦY CÓ THỂ HỎI VÀ CÁCH TRẢ LỜI
================================================================================

Câu hỏi 1: "Tại sao không dùng 1 database chung mà chia ra nhiều schema?"
Trả lời: Kiến trúc Modular Monolith theo DDD. Mỗi module = 1 Bounded Context
         có schema riêng, DbContext riêng. Lợi ích:
         - Module độc lập, không ảnh hưởng lẫn nhau
         - Dễ tách thành microservice khi cần scale
         - Team khác nhau có thể làm việc trên module khác nhau

Câu hỏi 2: "Tại sao không có FK giữa các module?"
Trả lời: Trong Modular Monolith, FK liên module tạo COUPLING ở DB level.
         Nếu có FK, không thể tách module ra database riêng sau này.
         Thay vào đó, validate ở application layer bằng service calls.

Câu hỏi 3: "Tại sao dùng UUID thay vì auto-increment?"
Trả lời: UUID có thể generate ở application layer mà không cần query DB.
         Phù hợp distributed system, tránh xung đột ID, không lộ số lượng
         record.

Câu hỏi 4: "Chi phí dùng AI như thế nào?"
Trả lời: Module LlmAssistant track từng lần gọi AI (ModelUsed, TokensUsed).
         Có cache layer (LlmSuggestionCaches) để giảm số lần gọi API.
         Module Subscription có UsageTrackings đếm LlmCallCount theo tháng
         và PlanLimits để giới hạn quota.

Câu hỏi 5: "Outbox Pattern là gì, tại sao cần?"
Trả lời: Đảm bảo event không bị mất khi gửi giữa các module.
         Lưu event vào DB cùng transaction với data -> background worker
         đọc và publish -> nếu crash thì retry từ DB.
         Pattern chuẩn trong Event-Driven Architecture.
