# API Test Generation Fix Report

## Current Flow Summary

OpenAPI parsing starts in `OpenApiSpecificationParser`, then `ParseUploadedSpecificationCommand` and `UploadApiSpecificationCommand` persist normalized endpoints, parameters, responses, security metadata, schemas, and examples. Endpoint metadata is later read through `ApiEndpointMetadataService` and `ApiEndpointParameterDetailService`.

SRS requirements are loaded in `GenerateLlmSuggestionPreviewCommand` and passed to `LlmScenarioSuggester`; unified callback generation loads them in `TestGenerationPayloadBuilder`. Before this fix, the LLM received broad SRS context and could attach unrelated requirements to an endpoint.

Endpoint prompts are built by `LlmScenarioSuggester`, `EndpointPromptContextMapper`, `ObservationConfirmationPromptBuilder`, and `TestGenerationPayloadBuilder`. Scenarios are generated by n8n/LLM, parsed in `LlmScenarioSuggester.ParseScenarios`, repaired with `ContractAwareRequestSynthesizer`, and supplemented by `EnsureAdaptiveCoverage`.

Ordered dependencies are computed by `ApiTestOrderAlgorithm` and `DependencyAwareTopologicalSorter`. Persisted test cases are enriched by `GeneratedTestCaseDependencyEnricher` in some generation paths, but callback-based AI save did not apply that enrichment consistently.

Request data comes from `ContractAwareRequestSynthesizer`; response expectations come from `ExpectationResolver` and `ErrorResponseSchemaAnalyzer`. Scenario count limits were driven by method constants, including exact POST/PUT/PATCH targets and fallback padding.

## Required Fixes

- Add endpoint-scoped SRS mapping and use only direct/partial high/medium matches for coverage.
- Persist OpenAPI request content type and derive generated `bodyType` from it.
- Rewrite prompt policy so OpenAPI provides the structural contract and SRS provides business intent.
- Remove happy-path-only constraint filtering.
- Filter expected statuses to OpenAPI-documented responses.
- Avoid strict success assertions on optional response fields unless SRS explicitly requires them.
- Replace exact scenario count padding with adaptive up-to limits and deduplication.
- Enforce dependency-aware request data reuse before suggestions/test cases are persisted.
- Remove random suffix mutation from the n8n workflow documentation.

## Endpoint-SRS Mapping Strategy

Each endpoint is matched against SRS requirements using HTTP method, path tokens, operation id, summary/description, parameter names, request/response field names, and documented response descriptions. Requirements are matched using code, title, description, type, refined/testable constraints, assumptions, and ambiguities.

Matches are classified as `direct`, `partial`, `dependency`, or `none`, and confidence is `high`, `medium`, or `low`. Only direct/partial high/medium matches are valid for `coveredRequirementCodes` and persisted traceability. Dependency matches are setup context only.

## OpenAPI vs SRS Responsibility

OpenAPI is the structural contract for method, path, endpointId, parameters, headers, request content type, request field names/types, required fields, schema constraints, documented statuses, response body structure, and error schemas.

SRS is the business source for endpoint-relevant requirements, scenario intent, business-critical constraints, meaningful test data, semantic assertions, and requirement coverage.

If SRS is missing or irrelevant, generation uses OpenAPI only and leaves requirement coverage empty.

## Scenario Generation Strategy

Generate one HappyPath when an endpoint is executable. Add Boundary and Negative scenarios only when OpenAPI schema, documented responses, auth requirements, or mapped SRS constraints expose real failure or mutation surfaces.

Scenarios are deduplicated by endpoint, test type, mutated field, mutation category, expected statuses, and covered requirement codes. Optional success response fields are not strict presence assertions from OpenAPI alone.

## Dependency Handling Strategy

Endpoint order from the approved sequence is preserved. If a later endpoint depends on an earlier one, the producer happy-path scenario must appear first and expose request/response variables. Register/login flows reuse `{{registeredEmail}}` and `{{registeredPassword}}`, with generic `{{email}}` and `{{password}}` aliases also available.

Duplicate-value negative tests reuse the prior created value and never generate a fresh `{{tcUniqueId}}` value for the duplicate field.

## Scenario Count Strategy

GET and DELETE endpoints target up to 3 scenarios. POST, PUT, and PATCH endpoints target up to 10 scenarios. The minimum is one HappyPath when executable. Scenarios are never padded with duplicates or weak variants to hit the target.

## Risk Areas

- Adding `EndpointParameter.ContentType` changes the ApiDocumentation EF model and requires a migration.
- Existing n8n deployments may still run an older workflow; backend post-processing must still enforce the policy.
- Some SRS markdown constraints do not identify an endpoint; semantic mapping must be conservative to avoid false traceability.
